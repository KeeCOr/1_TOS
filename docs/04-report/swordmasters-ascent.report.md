### 3.2 Design (설계 단계)

정식 설계 문서는 없으나, 다음과 같은 기술적 설계가 구현됨:

#### 핵심 데이터 모델 (gameData.ts)
- **Character**: 능력치(strength/agility/armor), HP/MP/stamina, 장비, 인벤토리, 마법, 타이틀
- **EnemyIntent**: 적의 주 액션/서브액션 + 확률 기반 분포
- **TurnResult**: 거리/행/피해/주사위 결과 통합
- **DiceResult**: rolls(배열), mode(highest/sum), value(사용 값)

#### 주요 함수들
- `calcNewPositions()`: 거리 변화 계산 (방향 충돌 방지)
- `calcNewRow()`: 행 이동 처리 (1-3 경계)
- `isAttackHitByRow()`: 행 기반 적중 판정 (공격만 행 영향, 마법/아이템은 무시)
- `distanceBonus()`: 거리에 따른 데미지 배수 (근거리 공격 보너스, 원거리 마법 보너스)
- `getEffectiveStats()`: HP/stamina 비율에 따른 동적 능력치 페널티
- `getDiceCount()`: 능력치 비율로 주사위 개수 결정 (1-4)
- `rollWithAdvantage()`: 주사위 굴림 + 모드 판정
- `diceMultiplier()`: 주사위 결과를 데미지 배수로 변환
- `resolveTurn()`: 턴 전체 해결 (거리·행·주사위·카운터·데미지·메시지)

#### 하위 시스템
- **마법 쿨다운**: 층 진행에 따라 5층 이상 시 쿨타임 제거
- **MP 비용**: 층 10 이상 시 비용 감소 (10 → 8)
- **저장/로드**: localStorage 기반 3개 슬롯, 타임스탬프 포함
- **층별 유령**: 층별로 최대 5개 유령 저장, 다시 만나면 100% 소환

## 5. 기술적 의사결정

### 5.1 아키텍처 선택

#### 1. 게임 로직 분리 (gameData.ts)
**결정**: 모든 게임 로직을 별도 파일로 분리
**이유**: 
- UI와 로직의 명확한 분리
- 유닛 테스트 용이성
- 재사용 가능한 계산 함수들
- 835줄의 순수 TypeScript 함수/타입

**장점**: 
- 약 1000줄의 컴포넌트 UI 로직과 명확히 분리
- 게임 규칙을 한 곳에서 관리
- 타입 안정성 (40+ 인터페이스/타입)

#### 2. 주사위 시스템: 2가지 모드
**결정**: highest vs sum(합산 크리티컬)
**로직**:
```
sum 모드 = 모든 주사위가 4 이상
highest 모드 = 그 외
배수: highest는 [0.4, 0.6, 0.8, 1.0, 1.35, 1.8]×
배수: sum은 1.4× ~ 2.8× (선형 증가)
```
**이유**: 
- 일반적인 경우와 운이 매우 좋은 경우 구분
- sum 모드 확률: 1/6^n (n=주사위 개수)
- 3주사위 sum = 1/216 (~0.46%)

#### 3. 거리 기반 전투 구현
**결정**: 능력치 비율로 주사위 개수 결정
**공식**:
```
비율 >= 2.0  → 4개 주사위
비율 >= 1.4  → 3개 주사위  
비율 >= 0.9  → 2개 주사위
그 외        → 1개 주사위
```
**이유**: 
- 능력치가 높을수록 주사위 더 굴림
- 능력치 비율로 전투 균형 조정
- perfect 매칭 시 +1 주사위, miss 시 -1 주사위

#### 4. 행 기반 회피 메커니즘
**결정**: 공격만 행 영향, 마법/아이템은 행 무관
**로직**:
```
ActionType === '공격' → 같은 행에만 명중
ActionType === '마법 사용' → 행 무관
ActionType === '아이템 사용' → 행 무관
```
**이유**:
- 근접 공격은 행으로 회피 가능 (전술성)
- 마법/원거리는 행 무시 (범용성)
- 게임 밸런스: 이동 서브액션의 가치 증대

#### 5. localStorage 기반 저장
**결정**: 3개 수동 저장 슬롯, 자동 저장 없음
**자료 구조**:
```typescript
SaveState {
  phase, floor, player, enemy, intent,
  playerPos, playerRow, enemyPos, enemyRow,
  magicCooldown, stats, logs, legacy
}
```
**이유**:
- 브라우저 저장소 크기 제약 (보통 5-10MB)
- 수동 저장으로 명확한 체크포인트
- 3개 슬롯으로 충분한 선택지

#### 6. 레거시 유령 시스템
**결정**: 층별로 최대 5개 유령 저장, 100% 소환 규칙
**로직**:
```
1. 플레이어 패배 → addFloorGhost(floor, player)
2. ghost_f{floor}_{timestamp} ID 생성
3. 동일 층 진입 시 → floorGhosts[floor]에서 무작위 소환
4. 층당 최대 5개 (FIFO로 오래된 것 제거)
```
**이유**:
- 플레이어의 강한 패배 피드백
- 재플레이 가치 증대
- localStorage 크기 제어

#### 7. 마법 냉각 및 MP 비용 스케일링
**결정**: 층 진행에 따른 동적 조정
**공식**:
```
냉각시간: floor < 5 → 1턴, floor >= 5 → 즉시
MP 비용: floor < 10 → 10, floor >= 10 → 8
MP 재생: floor < 10 → 8, floor >= 10 → 10
```
**이유**:
- 초반(1-4층): 마법 사용 제한으로 기본 공격 강제
- 중반(5-9층): 마법 접근성 증대
- 후반(10층+): 마법 의존성 증가 가능
